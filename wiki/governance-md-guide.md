# Writing a GOVERNANCE.md for Open Source Projects

A `GOVERNANCE.md` file is a project's **constitution**: it defines how decisions are made,
who holds what responsibility, and how the community operates. It works as a decision
framework, a community-structure definition, and a transparency tool that builds trust.

Synthesized from a "Governance.md Guide" web page (source URL wasn't captured in the saved
copy; content is generic best-practice guidance).

## The 5 Ws (and How)

- **Who — roles and people:** define authority and responsibility. Typical roles:
  **maintainers** (core team with commit access and decision-making authority),
  **contributors** (submit code/docs/improvements), and a **steering committee**
  (leadership for strategic decisions and conflict resolution).
- **What — scope and decisions:** which decision types need governance and what process
  applies to each, from small code changes to major direction shifts.
- **When** — the timing/triggers for governance processes (e.g. when a vote is required,
  review windows).
- **Where** — where decisions and discussions happen (issues, RFCs, meetings).
- **Why** — the purpose: clarity, structure, and trust for contributors and users.
- **How** — the concrete decision-making mechanism (lazy consensus, voting thresholds,
  escalation paths, conflict resolution).

The point: make the implicit rules explicit so contributors know how to participate and
how the project is steered.
